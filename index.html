The I-EPOS Project

I-EPOS is open source and accessible on https://github.com/epournaras/EPOS. The experimental live version is currently a branch of the I-EPOS repository and is accessible via https://github.com/epournaras/EPOS/tree/LiveExperiment The live version runs each agent in a separate process and can therefore be run in a computer network.

Getting Started

  • Start the virtual machine with configuration file vm/I-EPOS.vmx and virtual disk vm/I-EPOS.vmdk.
    The virtual machine was created with VMWare Player.
  • Login to user "I-EPOS VM" with password "iepos"

Step-by-Step Instructions

The instructions assume that the VM is started, the user logged in and on the homescreen as described in Getting Started.

  1. We consider a bicycle sharing system implemented in a city. Stations are set up all across the city, providing space for several bicycles each. As users drive across the city they move bicycles between the different stations. We notice that over time some stations are lacking bicycles, while others run out of bicycle storage. We try to solve this problem with I-EPOS.

    1. Start I-EPOS from the homescreen.

    2. Select the dataset by clicking the "Browse" button right of the "Data Set Location", and selecting the directory "bicycle".

    3. We want the change in the number of bicycles stored in each station to be zero. Therefore, we leave the "Global Cost Location" at "zero.target".

    4. For now we are only interested in minimizing the global cost, so please set the "Local Cost Influence" to 0.

    5. Click the "Run" button to start the simulation.

    6. While I-EPOS is executing, it tries to find combinations of user plans that make the total change in the number of bikes (referred to as global response) as small as possible. When the result window opens, it shows in the top section how the algorithm managed to reduce the global cost from its initial guess over a course of multiple iterations. The global cost is the squared distance of the global response from the target vector, that we left at zero in this example.

      On the bottom left we see the global response after the first iteration. Each dimension in this plot represents the change in the number of bikes for one bicycle station. The initial guess of the algorithm shows that at the end of the observed period, a station is missing 16 bikes. If plans were selected at random (i.e. if users can travel however they like), that change would be even greater.

      On the bottom right we see the network of agents. Each agent that changed its selection compared to the previous iteration is marked with the color black.

    7. Navigate through the iterations using the buttons "Next" and "Previous" to see how I-EPOS manages to improve the global response step by step. Notice that in the end I-EPOS is able to stabilize all stations so far that they loose or gain one bicycle at most, compared to the 16 in the first iteration.

      Furthermore, the changes in the network show that only certain groups of agents change their selection in each iteration. This is a result of the way I-EPOS merges changes together. Some agents are told not to change their selections in order to not hurt the overall performance.

    8. Close the report window, so that you are back at the homescreen.

  2. We notice that the users don't like the suggestions our system makes. If the users could freely choose their plans, the global response would be as follows:

    In order to make the system more appealing to users, we have to consider their preferences, encoded in the local cost.

    1. Start I-EPOS from the homescreen.

    2. Select the bicycle dataset again as explained above.

    3. Set the "Local Cost Influence" to 0.9

    4. Click the "Run" button to start the simulation.

    5. In the result window we can see that the global response (bottom left) is already pretty similar to the user preferred result. However, I-EPOS manages to turn this initial solution that many users like into a solution with much lower global cost without sacrifycing on the local cost. The local cost is represented as dashed line in the plot at the top.

    6. Navigate through the iterations again using the buttons "Next" and "Previous" to see how I-EPOS manages to improve the global response step by step. Notice that the global response is not as stable as it was in our previous simulation, but the local cost is close to optimum at around 0.004.

File reference

datasets/ Contains sample datasets for I-EPOS.
datasets/bicycle/ The bicycle dataset. It contains data of a bicycle sharing system. Each user of this system is represented as an agent. The possible plans of each agent are encoded as a vector with values as the incoming minus the outgoing bicycles of a user in each station at a certain time slot. For example, a user traveling from station 1 to station 3 and from station 4 to station 3 has the following plan: (−1, 0, 2, −1, . . .). I-EPOS can select recommended stations for each user agent such that the number of bicycles among the stations remains balanced.
datasets/energy/ The energy dataset. It contains data about the electricity consumption of households. Each household is represented as an agent. The possible plans of each agent are encoded as a vector with each value being the electricity consumption at one point in time. I-EPOS can be used to select plans in a way that electricity demand peaks are reduced.
datasets/gaussian/ The gaussian dataset. It contains generated data. The possible plans of each agent are drawn from a multivariate standard normal distribution.
datasets/full_datasets.zip Contains versions of the bicycle and energy dataset with more agents.
manual.pdf A description how to use I-EPOS and how to extend the functionality by adding custom datasets and custom cost functions.
vm/ Contains the virtual machine configuration as well as its virtual hard disk.
conf/ Contains configuration files used by I-EPOS.
lib/ Contains the libraries used by I-EPOS.
bin/IEPOS.jar The I-EPOS executable. Executable with the command java -jar IEPOS.jar.
videos/ Contains videos showing different algorithms in action. They demonstrate how agents change their plan selections over time.
